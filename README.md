# 里山調査用 オフライン地図アプリ

里道・境界調査向けの **完全オフライン動作** 地図アプリ（PWA）。
地理院地図・地番レイヤーの表示、GPS現在地、ワンタップ定型文メモを、通信のない山林でも利用できます。
仕様書 `satoyama_offline_map_spec.md` に基づき実装しています。

---

## 1. 構成ファイル

```
satoyama_map/
├── index.html              アプリ本体（画面構成）
├── manifest.webmanifest    PWA定義（ホーム画面追加用）
├── sw.js                   Service Worker（アプリ本体をオフラインキャッシュ）
├── css/style.css           UI（引き算設計・48px以上タップ領域・状態の色分け）
├── js/
│   ├── db.js               IndexedDB層（タイル・メモのローカル保存）
│   ├── tilelayer.js        オフライン対応タイルレイヤー（保存済→IndexedDB / 未保存→ネット）
│   └── app.js              地図・GPS・保存・メモのメインロジック
├── vendor/                 地図ライブラリ Leaflet（ローカル同梱＝オフラインでも起動）
├── icons/                  アプリアイコン（180/192/512）
├── data/parcels/           地番レイヤー（分割GeoJSON群 + index.json）
│   ├── index.json          読込モード・各ファイルの範囲(bbox)一覧
│   └── r{行}_c{列}.geojson   メッシュ分割された地番ファイル
├── tools/
│   ├── split_parcels.py    巨大な地番GeoJSONをメッシュ分割＋軽量化（大容量向け・推奨）
│   └── prepare_parcels.py  小さな地番GeoJSONを正規化する簡易ツール
└── server.py               動作確認用の簡易ローカルサーバー
```

## 2. 機能（仕様書との対応）

| 仕様 | 実装 |
| :-- | :-- |
| 3.1 背景地図・地番表示 | 地理院標準地図 + `data/parcels/` の地番ポリゴン重畳（大容量はメッシュ分割しオンデマンド読込） |
| 3.2 オフライン事前保存 | 画面中央の**ファインダー枠**にエリアを収めて保存。ズーム15〜18を自動一括DL |
| 〃 過負荷防止 | 枠内タイルが上限(600枚)超で保存ボタンを無効化し「範囲が広すぎます」を表示 |
| 3.3 GPS現在地 | 青い丸＋精度円で追従表示。「現在地に戻る」ボタン常設 |

| 3.5 ワンタップ定型文 | 🔀分かれ道 / 🌲倒木あり / 💧水場 /✏️その他（自由入力） |

## 3. iPhone/iPadへの導入（運用）

1. ファイル一式を **HTTPS** で配信できる場所に置く（PWAのGPS・Service WorkerはHTTPS必須）。
   - 例: 社内/個人のWebサーバー、GitHub Pages、Netlify など。
2. iPhoneの **Safari** で配信URLを開く。
3. 共有メニュー →「**ホーム画面に追加**」。
4. ホーム画面のアイコンから起動（全画面のアプリとして動作）。
5. **電波のあるうちに**、調査予定エリアで「地図を保存」を実行してタイルを端末に保存。
6. 以降は機内モード／圏外でも、地図閲覧・GPS・メモが利用可能。

> Service Worker により、一度オンラインで開けばアプリ本体（HTML/JS/Leaflet等）は
> オフラインでも起動します。地図画像は「地図を保存」で保存した範囲のみオフライン表示されます。

### ローカルでの動作確認
```bash
cd satoyama_map
python3 server.py          # http://127.0.0.1:8766 で起動
```
（localhost はHTTPS扱いとなりPWA機能を確認できます）

## 4. 地番データの準備（重要）

実調査では法務省「登記所備付地図データ」等を **PCで事前にGeoJSON化** して読み込ませます
（仕様書3.1：端末で地図XMLを直接読み込むのは不可）。
**地番ファイルは大容量になりやすいため、メッシュ分割＋オンデマンド読込を推奨**します。

### 4-1. 大容量データ（市町村1ファイルなど）→ メッシュ分割＋オンデマンド（推奨）

1. 元データを **EPSG:4326（経度緯度）のGeoJSON** にする。
   - 既にWGS84のGeoJSONならそのまま使えます（法務省データの多くは公共座標→WGS84変換済み配布あり）。
   - Shapefile等の場合: `ogr2ogr -f GeoJSON -t_srs EPSG:4326 parcels_wgs84.geojson source.shp`
2. メッシュ分割＋軽量化（地番のみ保持・座標6桁丸め）を実行:
   ```bash
   python3 tools/split_parcels.py path/to/big.geojson \
       --outdir data/parcels --grid 0.02 --precision 6 \
       --chiban-key 地番 --keep 大字名 --min-zoom 15
   ```
   → `data/parcels/` に分割ファイルと `index.json` が生成されます。
3. 必要なら `js/app.js` の `DEFAULT_CENTER` / `DEFAULT_ZOOM` を対象エリアへ更新。

> **仕組み**: `index.json` の `mode:"ondemand"` を見て、アプリは**地図の表示範囲に重なる
> ファイルだけ**を都度読み込みます。`minZoom`（既定15）より広域では地番を非表示にし、
> 拡大すると該当メッシュのみ読み込むため、総量が何百MBでも端末は軽いまま動作します。
>
> **実績（参考）**: 佐伯市の筆データ 332MB / 27万筆 を grid0.02・6桁丸めで分割 → 276ファイル・
> 合計163MB（最大3MB）。表示時に読むのは通常 1〜9ファイル（数MB）のみ。

### 4-2. 小規模データ → 単一ファイル

エリアが狭く軽量なら分割不要です。`tools/prepare_parcels.py` で地番プロパティを
`chiban` に正規化し、`data/parcels.geojson` として置けば読み込まれます
（`data/parcels/index.json` が無い場合の後方互換動作）。
```bash
python3 tools/prepare_parcels.py input.geojson --chiban-key 地番 --out data/parcels.geojson
```

### 4-3. 注意点

- **元の巨大GeoJSON（数百MB）は配信フォルダ(`data/`直下)に置いたままにしない**でください。
  分割後は不要です。Webへ載せると無駄な転送が発生します（分割後の `data/parcels/` だけ配信）。
- **地番のオフライン表示**は、地図タイル保存と同様に「一度オンラインで表示した範囲」が
  Service Workerに自動キャッシュされます。**現地に行く前に、調査エリアを一度表示**しておくと、
  地図保存とあわせて地番もオフラインで見られます。

### 4-4. 既知の制約: 地番が表示されない区画（山地・海域）について

`data/parcels/index.json` の分割タイルの中には、周囲を筆データに囲まれているにも
かかわらず、そのメッシュだけ地番が1件も存在しない「空白区画」が複数あります。
アプリでその範囲を表示すると、地番ポリゴンが描画されず地図が空白に見えます。

**これは `split_parcels.py` のバグやデータの取りこぼしではなく、元データ
（法務省「登記所備付地図データ」）にその区画の地番情報が存在しないことによる
自然な空白**であると調査の結果確認しました（判断根拠は末尾参照）。主な要因:

- 山頂・尾根など、地籍調査上そもそも地番が付与されていない、または山林の
  代表点（座標）が隣接メッシュ側に位置する土地
- 海域・湾内など、私有地番が存在し得ない区域

代表例: **`r1648_c6594`（臼坪山 山頂部、標高211m、佐伯市街地に囲まれた独立峰）**
以北で地番が表示されないのはこのためです。

#### 確認済みの空白区画（9件）

| セル | 中心付近の座標 | 現地確認結果 |
| :-- | :-- | :-- |
| r1648_c6594 | 32.970, 131.890 | 臼坪山 山頂部（標高211m、市街地に囲まれた独立峰） |
| r1638_c6583 | 32.770, 131.670 | 深山の尾根（標高674.5m、登山道のみ） |
| r1642_c6595 | 32.850, 131.910 | 山林（標高195.2m、建物・道路なし） |
| r1644_c6579 | 32.890, 131.590 | 旗返峠付近の深山（標高599.9m） |
| r1644_c6596 | 32.890, 131.930 | 石草峯付近の山林（標高123.2m） |
| r1643_c6600 | 32.870, 132.010 | 海域（沖合） |
| r1645_c6601 | 32.910, 132.030 | 海域（米水津湾） |
| r1645_c6596 | 32.910, 131.930 | ⚠️ 木立集落（標高21.3m、建物・道路あり）— **未解明・要再調査** |
| r1650_c6594 | 33.010, 131.890 | ⚠️ 中ノ内集落（標高51.7m、建物・道路あり）— **未解明・要再調査** |

> `r1645_c6596` と `r1650_c6594` の2件は、山地・海域のいずれにも該当せず、
> 実際には建物・道路のある谷沿いの集落でした。「山地・海域による自然な空白」の
> 説明が当てはまらないため、原因は別途調査が必要です。

#### 判断根拠

- 元の分割元データ（332MB・27万筆規模、リポジトリ非同梱）への直接アクセスはできない
  ため、以下の状況証拠から総合的に判断しました。
- `split_parcels.py` に筆数閾値によるスキップ処理は存在しない（コード確認済み）。
- `index.json` の合計筆数 272,670 は README記載の元データ規模「27万筆」と矛盾しない
  （大規模な取りこぼしがあれば概数はもっと大きくなるはず）。
- 空白区画同士はグリッド上で互いに隣接しておらず、最大37km離れて散在している。
  データ変換のバッチ処理（大字単位等）が丸ごと失敗したなら隣接する複数区画が
  まとまって欠けるはずで、1区画ずつ孤立する現在のパターンとは整合しない。
- 地理院地図で現地確認した9件中7件は山頂・尾根・海域と一致した（上表参照）。

## 5. 主な設定値（`js/app.js` 冒頭）

| 定数 | 既定 | 意味 |
| :-- | :-- | :-- |
| `SAVE_ZOOMS` | `[15,16,17,18]` | 一括保存するズームレベル |
| `MAX_TILES` | `600` | 1回の保存上限枚数（超で保存ブロック） |
| `WARN_TILES` | `450` | 枚数警告ライン |
| `DEFAULT_CENTER` / `DEFAULT_ZOOM` | 京都付近 / 15 | 初期表示位置 |
| `GSI_URL` | 地理院標準地図 | 背景タイルURL。淡色地図等へ変更可 |

## 6. 制約・運用上の注意（仕様書5）

- **GPS精度**: 谷底・樹冠下では数m〜数十mの誤差、または更新されない場合があります。
  ステータスは「GPS良好／誤差±○m／取得失敗」を色とテキストで表示します。
- **データ消失リスク**: メモ・保存地図は端末内(IndexedDB)のみに保存されます。
  Safariの履歴全削除や端末紛失で失われます。**恒久記録が必要な場合は調査後に
  スクリーンショット等で控える**運用ルールでカバーしてください。
- サーバー送信は一切行いません（メモは完全ローカル完結）。
