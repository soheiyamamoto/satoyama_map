# 里山調査用 オフライン地図アプリ

里道・境界調査向けの **完全オフライン動作** 地図アプリ（PWA）。
地理院地図・地番レイヤーの表示、GPS現在地、ワンタップ定型文メモを、通信のない山林でも利用できます。
仕様書 `satoyama_offline_map_spec.md` に基づき実装しています。

---

## 3. iPhone/iPadへの導入（運用）

1. ファイル一式を **HTTPS** で配信できる場所に置く（PWAのGPS・Service WorkerはHTTPS必須）。
   - 例: 社内/個人のWebサーバー、GitHub Pages、Netlify など。
2. iPhoneの **Safari** で配信URLを開く。
3. 共有メニュー →「**ホーム画面に追加**」。
4. ホーム画面のアイコンから起動（全画面のアプリとして動作）。
5. **電波のあるうちに**、調査予定エリアで「地図を保存」を実行してタイルを端末に保存。
6. 以降は機内モード／圏外でも、地図閲覧・GPS・メモが利用可能。

> Service Worker により、一度オンラインで開けばアプリ本体（HTML/JS/Leaflet等）は
> オフラインでも起動します。地図画像は「地図を保存」で保存した範囲のみオフライン表示されます。

```bash
cd satoyama_map

```

## 4. 地番データの準備（重要）

実調査では法務省「登記所備付地図データ」等を **PCで事前にGeoJSON化** して読み込ませます
（仕様書3.1：端末で地図XMLを直接読み込むのは不可）。
**地番ファイルは大容量になりやすいため、メッシュ分割＋オンデマンド読込を推奨**します。

### 4-0. 分割元データの所在（`data/source/`）

`data/parcels/` の分割元となった生データは `data/source/44205__2_r_2025.geojson`
に配置しています（佐伯市・公共座標2系・筆R・2025年版、332MB・27万筆規模）。

- **リポジトリには含まれません**（`.gitignore` で `data/source/` を除外）。
  肥大化を避けるため、GitHub上にはpushされていません。新しい開発環境で
  分割スクリプトを再実行する場合は、この元データを別途配置してください。
- **用途**: `tools/split_parcels_v2.py` で `data/parcels/` へ分割する際の入力データ。
  今後の里道抽出（筆の隙間の解析）やメッシュの切り直し（grid再生成）を行う際も、
  このファイルがソースとなります。

### 4-1. 大容量データ（市町村1ファイルなど）→ メッシュ分割＋オンデマンド（推奨）

1. 元データを **EPSG:4326（経度緯度）のGeoJSON** にする。
   - 既にWGS84のGeoJSONならそのまま使えます（法務省データの多くは公共座標→WGS84変換済み配布あり）。
   - Shapefile等の場合: `ogr2ogr -f GeoJSON -t_srs EPSG:4326 parcels_wgs84.geojson source.shp`
2. メッシュ分割＋軽量化（地番のみ保持・座標6桁丸め）を実行:
   ```bash
   python3 tools/split_parcels_v2.py path/to/big.geojson \
       --outdir data/parcels --grid 0.02 --min-grid 0.005 --max-count 1500 \
       --precision 6 --chiban-key 地番 --keep 大字名 --min-zoom 15
   ```
   → `data/parcels/` に分割ファイルと `index.json` が生成されます。
3. 必要なら `js/app.js` の `DEFAULT_CENTER` / `DEFAULT_ZOOM` を対象エリアへ更新。

> **仕組み（v2 / 現行）**: セル割り当ては「筆ポリゴンの外接矩形(bbox)が重なる全セル」に
> 出力する方式（1筆が複数セルにまたがる場合は該当する全ファイルに含め、重複を許容する）。
> 固定グリッドではなく quadtree式の可変分割で、1セルの筆数が `--max-count`（既定1500）を
> 超えると自動的に4分割（grid/2）し、再帰の下限は `--min-grid`（既定0.005 = grid/4）。
> `index.json` の各ファイルエントリは自身の `grid` 値を持つ（旧来のトップレベル単一`grid`は廃止）。
>
> `index.json` の `mode:"ondemand"` を見て、アプリは**地図の表示範囲に重なる
> ファイルだけ**を都度読み込みます。`minZoom`（既定15）より広域では地番を非表示にし、
> 拡大すると該当メッシュのみ読み込むため、総量が何百MBでも端末は軽いまま動作します。
>
> **実績（参考）**: 佐伯市の筆データ 332MB / 27万筆 を `--max-count 1500` で分割
> → **610ファイル**（v1のfirst_point方式・固定grid0.02では276ファイル。最大筆数は
> v1の6,993筆から1,487筆まで低減）。表示時に読むのは通常 数十ファイル程度。
>
> **旧方式（v1・`tools/split_parcels.py`）からの変更点**: v1は筆ポリゴンの
> 「最初の頂点」だけでセルを決めていたため、1筆が複数メッシュにまたがる場合に
> 境界の反対側からは地番が欠落して見えることがあった（v1は履歴として残しているが
> 現在は未使用）。
>
> **旧データのバックアップ**: 置き換え前の `data/parcels/`（v1・276ファイル）は
> `data/parcels_v1_backup/` にそのまま残してある。すぐに切り戻せるよう当面保持し、
> v2運用が安定してから削除する（削除時期は別途判断）。

### 4-2. 小規模データ → 単一ファイル

エリアが狭く軽量なら分割不要です。`tools/prepare_parcels.py` で地番プロパティを
`chiban` に正規化し、`data/parcels.geojson` として置けば読み込まれます
（`data/parcels/index.json` が無い場合の後方互換動作）。
```bash
python3 tools/prepare_parcels.py input.geojson --chiban-key 地番 --out data/parcels.geojson
```

### 4-3. 注意点

- **元の巨大GeoJSON（数百MB）は配信フォルダ(`data/`直下)に置いたままにしない**でください。
  分割後は不要です。Webへ載せると無駄な転送が発生します（分割後の `data/parcels/` だけ配信）。
- **地番のオフライン表示**は、地図タイル保存と同様に「一度オンラインで表示した範囲」が
  Service Workerに自動キャッシュされます。**現地に行く前に、調査エリアを一度表示**しておくと、
  地図保存とあわせて地番もオフラインで見られます。

### 4-4. 既知の制約: 地番が表示されない区画について

以下は v1（`tools/split_parcels.py`、固定grid・first_point方式）のデータで行った
調査の記録です。v2（現行、bbox全セル出力・quadtree可変分割）への切り替えにより、
下表のうち**4区画（r1642_c6595, r1643_c6600, r1644_c6579, r1644_c6596）は
隣接区画の筆がbboxの重なりにより複製され、ごく少数（1〜3筆）表示されるように
なりました**。残る5区画（r1638_c6583, r1645_c6596, r1645_c6601, r1648_c6594,
r1650_c6594）は、v2でも該当する筆が元データに存在しないため引き続き空白です。

`data/parcels/index.json` の分割タイルの中には、周囲を筆データに囲まれているにも
かかわらず、そのメッシュだけ地番がほとんど（またはv1では1件も）存在しない
「空白区画」が複数あります。アプリでその範囲を表示すると、地番ポリゴンが
描画されず地図が空白（または非常にまばら）に見えます。

**これはデータ生成スクリプトのバグやデータの取りこぼしではなく、元データ
（法務省「登記所備付地図データ」）にその区画の地番情報がほとんど存在しないことによる
空白**であると調査の結果確認しました（確認できた事実は末尾参照）。

理由は区画ごとに異なり、山頂・尾根・海域・低密度の山間部などが含まれますが、
**個々の区画で「なぜ元データに地番が無いのか」（地籍調査未了・国有林・
河川区域など）までは、手元のデータからは確定できません**。断定を避け、
現地で確認できた地形の事実のみを記載します。

代表例: **`r1648_c6594`（臼坪山 山頂部、標高211m、佐伯市街地に囲まれた独立峰）**
以北で地番が表示されないのはこのためです。

#### 確認できた事実（判断根拠）

- 元の分割元データ（332MB・27万筆規模、リポジトリ非同梱）への直接アクセスはできない
  ため、以下の状況証拠から総合的に判断しました。
- `split_parcels.py` に筆数閾値によるスキップ処理は存在しない（コード確認済み）。
- `index.json` の合計筆数 272,670 は README記載の元データ規模「27万筆」と矛盾しない
  （大規模な取りこぼしがあれば概数はもっと大きくなるはず）。
- 空白区画同士はグリッド上で互いに隣接しておらず、最大37km離れて散在している。
  データ変換のバッチ処理（大字単位等）が丸ごと失敗したなら隣接する複数区画が
  まとまって欠けるはずで、1区画ずつ孤立する現在のパターンとは整合しない。
- 地理院地図で現地確認した9件は、上表のとおりそれぞれ異なる地形だった。
- `split_parcels.py` は筆の最初の頂点座標だけでファイルを振り分ける仕様のため、
  「区画に重なる巨大な筆が、実際は隣接ファイル側に格納されているだけではないか」
  という仮説を検証した。全276ファイル・272,670筆を対象に「ポリゴンの外接矩形が
  その区画に少しでも重なる筆」を探索した結果:
  - 5区画（r1638_c6583, r1645_c6596, r1645_c6601, r1648_c6594, r1650_c6594）は
    該当する筆が**1件もヒットしなかった**。
  - 残る4区画（r1642_c6595, r1643_c6600, r1644_c6579, r1644_c6596）は、隣接ファイルの
    筆の外接矩形がわずかに境界を越えて重なっていたが、その筆自体は隣接区画側に
    格納されており、v1ではこの区画自体にポリゴンが存在しなかった。
  - いずれの場合も「この区画に本来属すべき筆が、別ファイルに紛れて実在する」という
    仮説（＝first_point方式の副作用）は支持されず、v1のバグではなく元データの
    性質によるものと判断した。ただしこの4区画については、境界越境自体は事実として
    確認できたため、後日 v2（bbox全セル出力方式）へ切り替えた際に実際に解消され、
    上表の「v2での状態」列のとおり少数の筆が表示されるようになった。

## 6. 制約・運用上の注意（仕様書5）

- **GPS精度**: 谷底・樹冠下では数m〜数十mの誤差、または更新されない場合があります。
  ステータスは「GPS良好／誤差±○m／取得失敗」を色とテキストで表示します。
- **データ消失リスク**: メモ・保存地図は端末内(IndexedDB)のみに保存されます。
  Safariの履歴全削除や端末紛失で失われます。**恒久記録が必要な場合は調査後に
  スクリーンショット等で控える**運用ルールでカバーしてください。
- サーバー送信は一切行いません（メモは完全ローカル完結）。

## 7. iOS PWA表示に関する既知の対応

### 画面下部がグレーに切れる問題（対応済み）

**症状**: iOS PWA（ホーム画面から起動したスタンドアロン表示）で、画面下部に
Safariアドレスバー相当の高さのグレーの帯が常時表示される。

**原因**: `apple-mobile-web-app-status-bar-style="black-translucent"` モードでは、
iOSがビューポートの高さを実際の画面全高より小さく報告し続ける挙動があった。
実機での数値確認により、`window.innerHeight` / `visualViewport.height` /
`document.documentElement.clientHeight` / `#map` の `clientHeight` は
すべて一致していた（793px）にもかかわらず、iPhone 15 の実際の画面高
（852px）との間に59pxの差があった。この差はホームバー分（約34px）だけでは
説明できない大きさで、`black-translucent` モード特有の挙動と判断した。

`map.invalidateSize()` の呼び出しタイミング追加、CSSの `100dvh` への切替、
`window.visualViewport` APIによる `#map` の高さ直接同期——いずれの対策でも
793pxという報告値自体は変わらなかった。つまりコード側の「高さの反映ロジック」
は正しく機能していたが、**iOSが返す値自体が実態と異なっていた**ため、
アプリ側の対策では解決しなかった。

**対応**: `index.html` の `apple-mobile-web-app-status-bar-style` を
`black-translucent` から **`default`** に変更（`js/app.js` の高さ同期処理・
CSSの `100dvh` フォールバックは残置）。`default` モードではステータスバー
領域がアプリの表示領域から明示的に除外され、ビューポートの報告値が実態と
一致するようになり、グレーの帯が解消した。

**トレードオフ**: ステータスバー裏までコンテンツが回り込む「全画面感」は
失われ、ステータスバー部分に明るい背景色の帯が見えるようになる。実機確認の
結果、この見た目の変化は許容範囲と判断した。

> 同じ症状が別の端末・iOSバージョンで再発した場合、まずこの`black-translucent`
> 由来のビューポート報告のズレを疑うこと。`window.innerHeight` 等の値と
> 実際の画面解像度を比較し、ホームバー分では説明できない差があれば
> 同一原因の可能性が高い。

## 8. アプリの自動更新とお知らせ

**背景**: 現地調査ではPWAを開きっぱなしのまま使う運用が多く、Service Workerの
標準動作（既存タブ/PWAが全て閉じられるまで新SWが有効化されない）のままでは
新しいバージョンが事実上ずっと反映されなかった。`sw.js` は `skipWaiting()` /
`clients.claim()` により新SWを即座に有効化する構成になっており、`js/app.js`
側でその有効化を検知して**控えめな更新バナー**（「新しいバージョンがあります。
タップで更新」）を表示する。反映（`location.reload()`）はバナーのタップ時、
または保存中・シート表示中でなければアプリ再開時（`visibilitychange` で
画面が表示状態に戻った時）に行い、作業中に突然リロードすることはない。

これとは別に、`version.json` を情報源として**「新しくなりました」お知らせ
モーダル**を用意している。起動のたびに前回見たバージョン（`localStorage`の
`lastSeenVersion`）と`version.json`の`version`を比較し、異なっていれば
平易な日本語の変更点を1回だけ表示する。初回起動時（`lastSeenVersion`未設定）
は記録のみ行い、モーダルは出さない。

### リリース時のチェックリスト

- **`version.json`を毎回更新する**: `version`（例: `2026.07.18`、同日複数回
  リリースする場合は`2026.07.18.2`のように連番を付ける）・`date`（人間向けの
  日付表示）・`notes`（3〜5行程度、専門用語を避けた平易な変更点）を更新する
- `sw.js`の`CACHE`名（例: `satoyama-shell-v3`）は**version.jsonとは同期させない**。
  こちらはキャッシュの持ち方自体（構造）を変えた時だけ手動で上げるものであり、
  毎回のコンテンツ更新では変更不要

- 地番: 法務省「登記所備付地図データ」等を変換して使用
- 地図描画: Leaflet（同梱）
